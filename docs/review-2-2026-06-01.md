## ⚠️ Headline: tests have been failing in CI on every commit since they were added

CI history (most recent first):

| Commit | Tests workflow | Docker workflow |
|---|---|---|

| `8f5c076` | ❌ failure | ✅ success |
| `5fde004` | ❌ failure | ✅ success |
| `dea67b9` | (first commit with tests) | … |

Four `fix:` commits explicitly tried to make the tests green
(`fix: correct indentation`, `fix: sqlite ALTER TABLE`,
`fix: correct refresh icon`, `fix: SVG icons`). None succeeded.

**The Docker workflow keeps publishing the image on every push
because it's a separate workflow that doesn't gate on Tests.**
So tickertape is currently shipping with broken tests AND a green
Docker badge. Verified locally:

```
TICKERTAPE_DB=/tmp/x.db pytest tests/
1 failed, 1 passed in 0.06s
```

Both failure modes are real:

1. **`test_idempotency_returns_same_id` with `:memory:`** — fails
   with `no such table: queue`. `init_db()` opens a fresh
   `sqlite3.connect()` per call. `:memory:` databases are
   per-connection, so the schema created in the fixture vanishes
   the moment `enqueue_job()` opens its own connection.
2. **`test_reminders_default_to_inbox` with a file DB** — fails
   with isolation problems. The first test's leftover row
   contaminates the second test.

**Root cause:** `db.py` uses a fresh `sqlite3.connect()` per
function call. With a file DB this survives across calls; with
`:memory:` it doesn't.

**Fix options:**
- Switch the fixture to a `tmp_path / "test.db"` file with
  explicit teardown (simplest), OR
- Refactor `db.py` to accept an injected connection / take a
  per-test connection-pool argument (more flexible).

## New issues I found while reviewing

### N1: Stray empty `resolving` file at repo root

```
?? resolving
```

Zero bytes, untracked. Looks like an accidental `> resolving`
shell redirection or a misnamed `git add`. Should be deleted.

### N2: Indentation in `db.py:init_db()` is funky but accidentally works

Lines 26-33 of the CREATE TABLE statement are under-indented (12
cols instead of 16). It's inside a `"""..."""` string, so SQL
doesn't care, but it's visually confusing — looks like the schema
falls out of the `with` block. The ALTER-TABLE migration block
right below (lines 37-48) is over-indented relative to the `def`.
Worth a 30-second whitespace pass.

### N3: `sanitize_text()` still lives in `worker.py`

The 95-line `EscPosReceipt` was deleted (good), but the 12-line
`sanitize_text()` function survived and is now applied **before**
strings are passed to upstream `Receipt`. Per round-1, this is
the right call (upstream doesn't have it; tickertape adds value
as a pre-processing layer). **Consider proposing this upstream**
as a `sanitize=True` kwarg on `Receipt`, since other downstream
apps probably want it too — would let tickertape drop the
wrapper.

### N4: `worker.process_job` has dead `save_to_history` line

```python
save = payload.get('save_to_history', True)   # line 35, dead
...
save = payload.get('save_to_history', True)   # line 64, used
```

Delete line 35.

### N5: Worker crash leaves jobs stuck in `'processing'`

Job grab atomically flips the row to `'processing'`. If the
worker process is killed mid-`process_job()`, the row is stuck
in `'processing'` forever — no other worker will pick it up.
Suggest: on worker startup, sweep `'processing'` rows older
than N seconds back to `'pending'`. Low priority — only matters
under crash/kill scenarios, but those happen in production.

### N6: The `Tests` workflow doesn't gate `Docker Build and Publish`

Right now it doesn't matter (Docker workflow ignores test status
regardless), but **after the tests are fixed**, the Docker
workflow should add `needs: test` — or be merged into a single
combined workflow. Otherwise broken releases can still publish.

### N7: My `CODE_REVIEW.md` got committed

It was supposed to be uncommitted scratch. Now it's in git

triage — but maybe rename to `docs/review-2026-06-01.md` for
posterity, or delete it after this round-2 review is also
triaged.
